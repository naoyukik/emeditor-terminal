# GitHub Issue 作成ガイドライン

このプロジェクトにおける GitHub Issue の作成および運用方法について定義する。

## 1. タイトル (Title)

タイトルは以下の形式に従う。原則として日本語で記述し、接頭辞（type）は半角小文字とする。

`type: 日本語での簡潔な説明`

### type の種類
- `feat`: 新機能の追加
- `fix`: バグ修正
- `refactor`: リファクタリング（機能変更を伴わないコードの整理・改善）
- `docs`: ドキュメントの更新
- `test`: テストの追加・修正
- `chore`: ビルドプロセスやツール、依存ライブラリの更新など

## 2. 内容 (Body)

Issue の内容は、後から見返した際に意図が明確に伝わるよう、以下のセクションを含めることを強く推奨する。

### 背景・目的 (Background / Purpose)
なぜこの作業が必要なのか、現状の課題は何かを記述する。関連する設計指針やガイドライン（例：`rust_win32.md`）がある場合は、その旨を明記する。

### タスク (Tasks)
具体的に実施すべき作業をチェックリスト形式 (`- [ ]`) で記述する。

### ゴール (Goal)
どのような状態になれば、この Issue を「完了（Closed）」とみなすことができるかを定義する。

## 3. 運用ルール

- **適切な粒度**: 1つの Issue で扱う範囲を大きくしすぎない。作業が多岐にわたる場合は、複数の Issue に分割することを検討する。
- **リファクタリングの根拠**: `refactor` の Issue を作成する際は、どのコード規約やアーキテクチャ方針に基づいた変更なのかを明示する。
- **透明性**: 思考過程や途中の気付きがあれば、Issue のコメント欄に随時記録する。
