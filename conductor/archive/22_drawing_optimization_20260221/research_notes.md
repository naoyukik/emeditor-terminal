# Research Notes: Issue #22 描画の最適化とちらつき防止

## 1. Microsoft Learn 調査結果
GDI ダブルバッファリングとフリッカー防止に関するベストプラクティスを以下の通り整理した。

### ダブルバッファリングの手順
1.  **メモリDCの作成**: `CreateCompatibleDC(hdc)` を使用して、ウィンドウDCと互換性のあるメモリDCを作成する。
2.  **互換ビットマップの作成**: `CreateCompatibleBitmap(hdc, width, height)` を使用して、描画領域サイズのビットマップを作成する。
3.  **ビットマップの選択**: `SelectObject(h_mem_dc, h_bm)` でメモリDCにビットマップをセットする。
4.  **オフスクリーン描画**: すべての描画操作（`ExtTextOutW`, `InvertRect` 等）をメモリDCに対して行う。
5.  **一括転送**: `BitBlt(hdc, 0, 0, width, height, h_mem_dc, 0, 0, SRCCOPY)` でウィンドウDCへ描画結果をコピーする。
6.  **リソース解放**: `DeleteObject(h_bm)` および `DeleteDC(h_mem_dc)` を確実に行い、GDIオブジェクトのリークを防止する。

### フリッカー回避の重要ポイント
-   **WM_ERASEBKGND の抑制**: このメッセージで `1` (TRUE) を返すことで、Windowsによる背景の自動消去（白塗り）を防止する。これがダブルバッファリングと組み合わさることで、完全なフリッカーフリーが実現する。
-   **DWMとの兼ね合い**: 近代的なWindowsでは DWM がシステムレベルでバッファリングを行っているが、アプリケーション側で「背景消去→逐次描画」を行うと、依然として一瞬の空白（ちらつき）が視認されるため、アプリケーション側でのダブルバッファリングは有効である。

## 2. EmEditor SDK 調査結果
-   `sdk/` 配下のドキュメントおよびヘッダーファイルを確認したが、プラグインウィンドウ（カスタムバー）の描画を直接最適化する専用の API やメッセージは存在しない。
-   プラグインウィンドウは標準的な Win32 子ウィンドウとして動作するため、標準的な GDI 最適化手法を適用するのが正解である。

## 3. 実装への反映
-   `TerminalGuiDriver::render` 内でメモリDCを生成・破棄する「オンデマンド方式」を採用する。
-   リソース管理を厳格に行い、特にリサイズ時や描画失敗時のリークを防止する。
