# Specification: Grapheme Clusters 方式による文字処理エンジンの刷新

## 1. Overview
現在の `char` 単位のセル管理を、Unicode の **Grapheme Clusters (書記素クラスター)** 単位へと刷新する。これにより、結合文字、複雑な絵文字（Emoji ZWJ シーケンス）、および全角文字の混在環境下での正確なテキスト測定と描画を実現し、Issue #104 等の座標ズレ問題を根本から解決する。

## 2. Functional Requirements
- **書記素クラスターの認識**:
    - `unicode-segmentation` クレートを使用し、ユーザーが「1文字」と認識する最小単位を正しく抽出する。
    - 複数の `char` から成る結合文字や Emoji ZWJ シーケンスを1つの論理的な単位として扱う。
- **Cell 構造体の拡張**:
    - `Cell` は単一の `char` ではなく、可変長の `String` (またはそれに準ずるバッファ) を保持し、1つの書記素クラスターを完全に格納する。
- **正確な表示幅の算出**:
    - `unicode-width` クレートを使用し、クラスター全体の表示カラム数（1 or 2）を正確に算出する。
- **バッファリングと確定処理**:
    - 入力バイトストリームから不完全なクラスターを検知した際、確定するまで（または次の有効なクラスターが開始されるまで）描画を保留し、表示の整合性を保つ。
- **既存ロジックの刷新**:
    - 既存のバイトベース UTF-8 再構築ロジックを、`unicode-segmentation` を活用したストリーム処理へと統合・刷新する。

## 3. Non-Functional Requirements
- **正確性**: どんな複雑な Unicode シーケンスに対して、物理グリッド（カラム）と論理文字の対応が決してズレないこと。
- **堅牢性**: 不正なバイト列や未完成のシーケンスに対しても、クラッシュせず適切にフォールバックすること。
- **パフォーマンス**: 大量の文字出力時も、文字列アロケーションによるオーバーヘッドを最小限に抑える（必要に応じて `SmallVec` 等の検討も含む）。

## 4. Acceptance Criteria
- [x] 家族の絵文字や国旗などの Emoji ZWJ シーケンスが、1つの「文字」として正しく描画・削除（バックスペース）される。
- [x] 日本語の濁点・半濁点の結合文字形式 (NFD) が、バラバラにならず1つの全角文字として扱われる。
- [x] 全角文字（2カラム）の間に1バイト文字を挿入・削除した際、周囲の文字の座標が1カラム分ズレる現象（Issue #104）が完全に解消されている。
- [x] すべての既存の TUI 描画テストがパスすること。

## 5. Out of Scope
- 文字のグリフ（フォント）自体のレンダリング改善（OSの描画APIに依存するため）。
- RTL（右から左へ書く言語）の完全なサポート。
- 合字（リガチャ）の特殊な制御（フォントの機能に依存するため）。
